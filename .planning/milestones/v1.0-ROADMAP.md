# Milestone v1.0: DirectTalk MVP

**Status:** ✅ SHIPPED 2026-06-06
**Phases:** 1-3
**Total Plans:** 3

## Overview

A zero-dependency, LAN-only shared transcript for two or more Macs. The host Mac runs a
small `node:http` server; everyone (host included) opens a browser to read and append to a
common, append-only transcript. Built to replace hand-copying multi-line commands and
their output between a Claude Code session on one Mac and a terminal on another — no
accounts, no third-party sign-ons, zero install on client Macs.

## Phases

### Phase 1: Server & Session Persistence

**Goal**: A zero-dependency Node server runs on the host Mac, manages named sessions, and persists them to disk so they survive a restart.
**Depends on**: Nothing (first phase)
**Plans**: 1 plan

Plans:
- [x] 01-01: `sessions.js` persistence module + `node:http` server (create/list sessions, JSONL persistence, restart survival)

**Details:**
Requirements: SRV-01, SRV-02, SESS-01, SESS-02, SESS-03.
Built `sessions.js` (append-only JSONL store: collision-safe slug ids, atomic exclusive
create, path-traversal-safe id validation, ANSI-stripping `appendMessage`, tolerant
readers) and `server.js` (`node:http`, `PORT||5757`, `sessions/` bootstrap, route dispatch,
`/api/sessions` create+list, 400/500 resilience, 1MB body cap). Restart survival proven by
a test that spawns and SIGTERMs the real server process.

### Phase 2: Live Messaging & Text Fidelity

**Goal**: Text posted from any browser appears live in every connected browser, intact and correctly attributed, with terminal output rendered cleanly.
**Depends on**: Phase 1
**Plans**: 1 plan

Plans:
- [x] 02-01: SSE stream (replay-then-live) + POST messages (role detection, ANSI/multiline fidelity, broadcast) + LAN access doc

**Details:**
Requirements: CONN-01, MSG-01, MSG-02, MSG-03, MSG-04, TXT-01 (data layer).
Added `GET /api/sessions/:id/stream` (SSE: subscribe-before-snapshot to avoid a
message-dropping race, then replay stored transcript, then push live; per-message `seq`
for connect-boundary dedupe; heartbeats; dead-socket pruning) and
`POST /api/sessions/:id/messages` (Host/Client role by `remoteAddress`, append + broadcast).
Byte-faithful multi-line text end to end. Server binds all interfaces so `<host>.local`
resolves on the LAN.

### Phase 3: Browser UI

**Goal**: Users on any Mac can pick or create a session, read the live transcript, send messages, and see connection status — all in one browser page with zero install.
**Depends on**: Phase 2
**Plans**: 1 plan

Plans:
- [x] 03-01: `index.html` single-file UI (picker, transcript pane, entry box, connection status) + `GET /` static route on `server.js` + static test

**Details:**
Requirements: UI-01, UI-02, UI-03, UI-04 (plus the rendering half of TXT-01).
Built `index.html` as one self-contained file (HTML + inline CSS + inline JS, zero client
deps): session picker, scrollable monospace `white-space: pre-wrap` transcript rendered via
`textContent` (XSS-safe, never `innerHTML`), side + timestamp per line, pin-to-bottom
auto-scroll, textarea + Send with Cmd/Ctrl+Enter (clear-on-success / keep-on-failure), and
an `EventSource`-driven connection-status indicator (connecting / connected / reconnecting /
disconnected) with `seq` dedupe across reconnects. Served from the existing server at `GET /`.

## Milestone Summary

**Decimal Phases:** None.

**Key Decisions:**
- Browser form factor (host server + browser UI) — zero install/trust on family Macs.
- Pure Node, SSE + POST, zero npm dependencies — append-only transcript fits push-only SSE.
- No access barrier — trusted home network (documented, accepted trade-off).
- Two-side attribution (Host/Client) by connection origin — no role-picker UI.
- Append-only JSONL persistence — crash-safe, no rewrite races, trivial resume.

**Issues Resolved (caught by per-phase code review + fixed):**
- Path traversal via session id (Phase 1 hardening).
- Non-atomic session create / orphan meta-less files (Phase 1 hardening).
- SSE replay/live race that could permanently drop a message at connect (Phase 2).
- Concurrent-post `seq` collision that defeated client dedupe (Phase 3 review → fix).
- POST 404 TOCTOU, dead-socket leak, terminal-404 endless "reconnecting" (Phases 2-3).

**Issues Deferred:** None blocking.

**Technical Debt Incurred:**
- Per-request `fs.readFileSync` of `index.html` at `GET /` (accepted: tiny file, LAN-only).
- Test files share a server singleton (passes under per-file process isolation only).

**Verification:** 29/29 automated tests pass; live single-host smoke test green; two-Mac
LAN verification confirmed by the user (live sync, role labels, formatting fidelity,
reconnect/status, failure-path text retention).
