Phase 2 API the browser consumes:
- GET  /api/sessions
    -> 200 JSON: [{ id, name, createdAt }]   (sorted newest-first)
- POST /api/sessions          body { name }
    -> 201 JSON: { id, name, createdAt }      (400 if name missing/blank)
- GET  /api/sessions/:id/stream               (SSE, text/event-stream)
    -> replays stored transcript, then pushes live; each SSE frame is
       `data: <JSON.stringify(record)>\n\n`; comment heartbeats `:\n\n` every 15s
    -> 404 JSON if session id unknown
- POST /api/sessions/:id/messages   body { text }
    -> 201 JSON: the stored record           (400 if text empty; 404 if unknown id)

Message record shape (from sessions.js — both replay and live use this):
    { type: "msg", seq: <int>, side: "host" | "client", text: <string>, ts: <ISO string> }
  - `seq` is monotonic per session and IDENTICAL for replay and live frames at the
    connect boundary — dedupe on it so a reconnect's replay does not duplicate/lose lines.
  - `text` is already ANSI-stripped and may contain newlines/indentation — preserve it
    via CSS `white-space: pre-wrap` and render via `textContent` (never `innerHTML`).
  - `side` is server-stamped ("host" for loopback origin, else "client") — just display it.
  - Session id regex on the server is /^[a-z0-9-]+$/.

Existing server.js route dispatcher (server.js `handler`):
  - new URL(req.url, 'http://localhost'); checks `pathname` exactly.
  - Order: GET /api/sessions, POST /api/sessions, then regex match
    /^\/api\/sessions\/([a-z0-9-]+)\/(stream|messages)$/, then a trailing 404.
  - Helpers available: sendJson(res, status, obj), readBody(req).
  - server is exported; listens only under require.main===module (tests bind port 0).

  <action>
    Create `index.html` as ONE self-contained file: a single `<!doctype html>` document with
    inline `<style>` and inline `<script>`. ZERO external resources — no CDN links, no
    `<script src=...>`, no `import`, no React. Pure browser APIs (fetch, EventSource, DOM).

    Layout — two views toggled by JS (both in the one file):
    A) PICKER VIEW (UI-01):
       - On load, `fetch('/api/sessions')` -> render a list; each row shows the session
         `name` and the `createdAt` date formatted in the viewer's LOCAL time, compactly
         (e.g. `new Date(createdAt).toLocaleString()`). Clicking a row enters that session.
       - A "New session" control: prompt for a name (window.prompt is fine), then
         `fetch('/api/sessions', {method:'POST', headers:{'Content-Type':'application/json'},
         body: JSON.stringify({name})})`; on 201 enter the returned session id. On failure,
         show an inline error and stay on the picker.
       - Empty list must still render cleanly (e.g. an empty-state line), not crash.

    B) SESSION VIEW (UI-02, UI-03, UI-04 + TXT-01 rendering):
       - A "back to sessions" control that closes the EventSource and returns to the picker.
       - TRANSCRIPT PANE (UI-02 + TXT-01): a scrollable container with CSS
         `font-family: monospace;` and `white-space: pre-wrap;`. For each message render a
         line/element showing the `side` (display as `Host`/`Client`) and `ts` (local time,
         compact) as a prefix/label, plus the message body. Set the message body text via
         `el.textContent = rec.text` — NEVER `innerHTML` (XSS-safe + WYSIWYG). The prefix
         (side + timestamp) may use textContent too. AUTO-SCROLL: after appending, scroll the
         pane to the bottom; pin-to-bottom only when the user was already at/near the bottom
         (compute before appending: wasAtBottom = scrollTop + clientHeight >= scrollHeight - <small px>).
       - DEDUPE (CONTEXT decision): keep `let maxSeq = -1;` (per session). On each incoming
         record, if `rec.seq <= maxSeq` skip it (already shown); else render and set
         `maxSeq = rec.seq`. Reset maxSeq to -1 when entering a session / opening a new stream.
         This makes the connect-boundary replay-vs-live overlap idempotent across reconnects.
       - SSE WIRING (UI-02 live + UI-04): `const es = new EventSource('/api/sessions/'+id+'/stream');`
         `es.onmessage = (e) => { const rec = JSON.parse(e.data); /* dedupe + render */ };`
         Heartbeat comment frames produce no `message` event, so no special handling needed.
       - ENTRY BOX (UI-03): a `<textarea>` + a Send `<button>`. Sending posts
         `fetch('/api/sessions/'+id+'/messages', {method:'POST',
         headers:{'Content-Type':'application/json'}, body: JSON.stringify({text})})`.
         On 201 (ok): clear the textarea. On non-ok / network error: KEEP the text in the
         textarea and surface a visible error message. Keybind: Cmd+Enter AND Ctrl+Enter send
         (keydown where (e.metaKey || e.ctrlKey) && e.key === 'Enter'); plain Enter inserts a
         newline (do NOT send) so multi-line paste/compose works. Do not send empty text.
       - CONNECTION STATUS (UI-04): an always-visible indicator in the session view. Set it
         to "connected" on `es.onopen`, and "reconnecting" on `es.onerror` (EventSource
         auto-reconnects; on a successful reconnect onopen fires again -> back to "connected").
         Initial state before first open: "connecting"/"reconnecting" is fine.

    Keep CSS minimal and functional (no framework). Do not add a role picker — `side` comes
    from the server. Do not escape text manually anywhere — rely on textContent.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| message text -> browser DOM | Stored message text (from any LAN client) is rendered into the host/client browsers; an attacker-controlled payload could attempt script injection. |
| browser -> server static route | GET / reads a fixed file path (index.html); no user input forms the path. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering/Elevation (XSS) | transcript rendering in index.html | mitigate | Render all message text via `el.textContent` (never `innerHTML`); CSS `white-space: pre-wrap` shows markup literally. Grep gate `! grep -q innerHTML index.html` enforces it. |
| T-03-02 | Information Disclosure | GET / static route | accept | Serves only the fixed `index.html` (no path from user input, no traversal surface). Transcripts already readable on the trusted LAN per project scope. |
| T-03-03 | Denial of Service | GET / reads file per request | accept | LAN-only, tiny file, trusted users; per-request `readFileSync` is acceptable for the deployment scale. No mitigation warranted. |
</threat_model>
